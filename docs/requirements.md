## 1. 概要

### 1.1 システムの目的

食品流通システム（チッタ・ハンチョウ）は、中小規模の食品製造・卸売企業を対象とした受発注業務DX（デジタルトランスフォーメーション）システムである。FAX・紙伝票を中心とした従来の受発注プロセスを、AI-OCR技術とWebベースのワークフロー管理によってデジタル化し、業務効率の大幅な向上とヒューマンエラーの削減を実現する。

### 1.2 対象ユーザー

| 区分 | 対象 | 説明 |
|------|------|------|
| プライマリ | 事務担当者 | 日常的に受発注・在庫・請求業務を行う担当者（1〜3名規模） |
| セカンダリ | 管理職（課長・部長） | 承認業務、日次・月次レポートの確認 |
| セカンダリ | 倉庫担当者 | 出荷指示の受領、在庫の実数確認 |
| テリシャリ | 経営層 | 月次売上・損益の概況確認 |

### 1.3 背景と課題

高知県を拠点とする中小食品企業の多くは、依然として以下のようなアナログな業務フローに依存している。

- **受注**: 得意先からFAXで届く注文書を目視で確認し、手作業で基幹システムやExcelに転記
- **在庫管理**: 紙の在庫表やホワイトボードで管理。冷凍庫・冷蔵庫を実際に見に行かないと正確な数量がわからない
- **請求書**: Excelテンプレートに手入力で作成。転記ミスや計算ミスが月に数件発生
- **返品処理**: 電話連絡ベースで記録が残らず、後日の確認で齟齬が生じる

こうした課題を包括的に解決するため、受注〜出荷〜請求〜返品までを一気通貫で管理できるシステムとして食品流通システムを開発する。

### 1.4 システム構成概要

- **フロントエンド**: Next.js（App Router）/ React / Tailwind CSS
- **バックエンド**: Next.js API Routes / Prisma ORM
- **データベース**: PostgreSQL
- **AI-OCR**: 外部OCRサービス連携（将来的に自社モデルへ移行可能な設計）
- **会計連携**: freee / マネーフォワード等へのCSV/APIエクスポート

以下は、山田美咲さんの典型的な1日の業務フローである。食品流通システムの各機能がどの時間帯にどのように使われるかを具体的に記述する。

### 3.1 8:30 - 出社・ダッシュボード確認

#### 誰が
山田美咲（事務担当者）

#### 何を見て
出社後、PCを起動しブラウザで食品流通システムにログイン。最初に目に入るのは**ワークフローステップバー**（WorkflowStepBar）である。ステップバーは画面上部に常時表示され、6つのワークフローステップそれぞれに未処理件数がバッジで表示される。

山田さんが確認する情報:
- **受注（OCR読取）**: 未処理FAX/PDFの件数 → 例: 「8件」（昨夜〜今朝に届いたFAX）
- **発注確認**: OCR読取完了で確認待ちの件数 → 例: 「3件」（前日夕方に読み取ったが未確認のもの）
- **在庫確認**: 在庫引当待ちの件数 → 例: 「2件」
- **出荷処理**: 出荷指示待ちの件数 → 例: 「4件」（本日出荷予定分）
- **請求書発行**: 未発行の請求書件数 → 例: 「3件」
- **返品処理**: 対応待ちの返品件数 → 例: 「1件」

#### 何をするか
1. ワークフローステップバーの各バッジ件数を一目で確認する
2. 特に件数が多い箇所や、赤色のアラートが出ている箇所を優先的に把握する
3. 最も緊急度の高いステップ（通常は「受注」）をクリックして業務を開始する

#### 画面表示要件
- ワークフローステップバーは6ステップを横一列（デスクトップ）または縦一列（モバイル）で表示
- 各ステップには、ステップ番号（1〜6）、アイコン、タイトル、サブタイトル、未処理件数バッジを表示
- 現在アクティブなステップはリング（青色: `ring-2 ring-blue-500`）でハイライト
- 未処理件数が0件の場合はグレーバッジ（`bg-gray-100 text-gray-400`）、1件以上の場合はブルーバッジ（`bg-blue-100 text-blue-700`）
- 件数が10件を超える場合は赤色バッジ（`bg-red-100 text-red-700`）に切り替え、緊急性を視覚的に示す
- ステップ間は矢印（`→`）で接続し、業務の流れを直感的に表現

#### 操作性要件
- ダッシュボード表示は1秒以内にレンダリング完了すること
- ワークフローステップバーの各ステップをクリックすると、該当ページに直接遷移すること（1クリックで到達）
- キーボードショートカット: `Alt + 1`〜`Alt + 6` で各ステップページへ直接遷移

### 3.2 9:00 - 受注処理（OCR読取）

#### 誰が
山田美咲（事務担当者）

#### 何を見て
`/ocr` ページ（AI-OCR 伝票読取画面）を開く。

この画面には以下の情報が表示される:
- **画像アップロードエリア**: ドラッグ&ドロップまたはクリックでFAX画像（JPEG, PNG, GIF, WebP, TIFF）をアップロード
- **最近のスキャン履歴テーブル**: スキャンID、文書種別（納品書/請求書/発注書）、ステータス、信頼度、スキャン日時

山田さんの目の前にある具体的な注文:
- イオン高知店から: 抹茶アイスクリーム120ml × 200個、ゆずソルベ × 150個、土佐ジロー卵プリン × 100個
- ローソン四国エリアから: 塩けんぴアイスバー × 500本、よさこいマンゴーパフェ × 300個
- 道の駅やすから: 四万十栗モンブランアイス × 80個、ゆずソルベ × 60個

#### 何をするか
1. 複合機のスキャナから取り込んだFAX画像ファイルをPCのフォルダに保存する（またはメール添付PDFを保存）
2. `/ocr` ページのアップロードエリアにファイルをドラッグ&ドロップする
3. OCR処理が自動的に開始される。ステータスが「PROCESSING」（青色バッジ、パルスアニメーション付き）に変わる
4. 処理完了後、ステータスが「COMPLETED」（緑色バッジ）に変わり、信頼度（例: 95%）が表示される
5. 信頼度が90%以上の場合は緑色、70〜90%は黄色、70%未満は赤色で表示
6. 「FAILED」（赤色バッジ）の場合は、画像の品質が悪い可能性がある。再スキャンまたは手動入力を検討
7. 1件ずつアップロードを繰り返し、全FAX（8件）の読み取りを完了させる

#### 画面表示要件
- アップロードエリア: 中央に配置、破線ボーダー（`border-2 border-dashed border-gray-300`）
- ドラッグ中はボーダーが青色に変化（`border-blue-500 bg-blue-50`）し、ドロップ可能であることを視覚的にフィードバック
- ファイル選択後、ファイル名とサイズを青色の背景バー（`bg-blue-50`）で表示
- スキャン履歴テーブルのカラム: スキャンID / 文書種別 / ステータス / 信頼度 / スキャン日時
- ステータスバッジ色:
  - PENDING: グレー（`bg-gray-100 text-gray-800`）
  - PROCESSING: 青色パルス（`bg-blue-100 text-blue-800 animate-pulse`）
  - COMPLETED: 緑色（`bg-green-100 text-green-800`）
  - FAILED: 赤色（`bg-red-100 text-red-800`）
- 信頼度表示: パーセンテージとカラーコード（90%超=緑、70〜90%=黄、70%未満=赤）

#### 操作性要件
- ファイルのドラッグ&ドロップは直感的に行えること（ドロップゾーンは十分な大きさ: `p-12`）
- 複数ファイルの連続アップロードに対応すること（1ファイルアップロード完了後、即座に次のファイルをドロップ可能）
- OCR処理中はプログレスインジケーターを表示し、処理が進行中であることを明示
- アップロードから読取完了まで最大30秒以内（A4サイズFAX画像1枚の場合）
- スキャン履歴は最新のものが上に表示され、最大50件まで一覧表示

### 3.3 10:00 - 発注確認・修正

#### 誰が
山田美咲（事務担当者）→ 必要に応じて鈴木課長（承認者）

#### 何を見て
`/orders` ページ（発注管理画面）を開く。

OCRで読み取られた注文データが、発注一覧テーブルに表示される。各行には以下の情報がある:
- **発注番号**: 自動採番（例: `ORD-20260318-001`）
- **発注元**: 注文を出した得意先名（例: イオン高知店）
- **発注先**: 高知アイス株式会社（自社）
- **ステータス**: 下書き / 発注済 / 確認済 / 出荷済 / 納品済 / キャンセル
- **金額**: 合計金額（例: ¥156,800）
- **発注日**: OCRで読み取った日付

具体例として、今朝OCR読取した注文:

| 発注番号 | 発注元 | ステータス | 金額 | 発注日 |
|---------|--------|-----------|------|--------|
| ORD-20260318-001 | イオン高知店 | 下書き | ¥385,000 | 2026/03/18 |
| ORD-20260318-002 | ローソン四国エリア | 下書き | ¥520,000 | 2026/03/18 |
| ORD-20260318-003 | 道の駅やす | 下書き | ¥89,600 | 2026/03/18 |
| ORD-20260317-008 | サニーマート全店 | 確認済 | ¥234,500 | 2026/03/17 |
| ORD-20260317-009 | フジグラン高知 | 出荷済 | ¥178,200 | 2026/03/17 |

#### 何をするか
1. ステータスが「下書き」（グレーバッジ `bg-gray-100 text-gray-700`）の発注を上から順に確認していく
2. 発注番号をクリックして詳細画面を開く
3. OCRが読み取った内容（商品名、数量、単価）を元のFAX画像と照合する
4. **修正が必要な場合**: 例えば、OCRが「抹茶アイスクリーム120ml × 200」を「抹茶アイスクリーム120ml × 2000」と誤読した場合、数量フィールドを手動で修正
5. **商品コードの確認**: OCRが商品名から自動マッピングした商品コード（例: PRD-101）が正しいか確認。誤りがあればプルダウンから正しい商品を選択
6. 確認が完了したら「確認完了」ボタンをクリック。ステータスが「下書き」→「確認済」（緑色バッジ `bg-green-100 text-green-700`）に変わる
7. 金額が10万円を超える注文、または通常と大きく異なる数量の注文は、鈴木課長に承認を依頼する（画面上の「承認依頼」ボタンをクリック）

#### 鈴木課長の操作（承認フロー）
- 山田さんから承認依頼が来ると、課長のダッシュボードに通知が表示される
- 注文内容を確認し、「承認」または「差戻し」をクリック
- 差戻しの場合はコメント（例: 「イオン高知店の抹茶アイスの単価、先月の契約単価と異なります。確認してください」）を入力

#### 画面表示要件
- 発注一覧テーブルのカラム: 発注番号（青色リンク） / 発注元 / 発注先 / ステータス（色分けバッジ） / 金額（右寄せ、`tabular-nums`） / 発注日
- ステータスバッジの色:
  - 下書き（DRAFT）: グレー（`bg-gray-100 text-gray-700`）
  - 発注済（SUBMITTED）: 青色（`bg-blue-100 text-blue-700`）
  - 確認済（CONFIRMED）: 緑色（`bg-green-100 text-green-700`）
  - 出荷済（SHIPPED）: 黄色（`bg-yellow-100 text-yellow-800`）
  - 納品済（DELIVERED）: エメラルド（`bg-emerald-100 text-emerald-700`）
  - キャンセル（CANCELLED）: 赤色（`bg-red-100 text-red-700`）
- テーブルフッターに合計件数と合計金額を表示
- 「新規発注」ボタン（青色、右上）で手動発注も可能
- サンプルデータ表示時はアンバー色の注意バナー（`bg-amber-50 border-amber-200`）を表示

#### 操作性要件
- 一覧から詳細画面への遷移は1クリック
- 詳細画面でのOCR結果修正は、該当フィールドをクリックして直接編集可能（インライン編集）
- 修正箇所はハイライト表示（黄色背景）され、変更履歴が残る
- 「確認完了」操作は最大3クリック以内（一覧→詳細→確認完了ボタン）
- ステータスフィルター機能: プルダウンで特定ステータスの発注のみ絞り込み表示
- 検索機能: 発注番号、得意先名、商品名でのインクリメンタルサーチ

### 3.4 11:00 - 在庫確認・引当

#### 誰が
山田美咲（事務担当者）、田中誠（倉庫担当・実在庫の確認）

#### 何を見て
`/inventory` ページ（在庫管理画面）を開く。

在庫一覧テーブルに、現在の全在庫が表示される:

| 商品名 | 商品コード | 数量 | 単位 | ロット番号 | 保管場所 | 賞味期限 |
|--------|-----------|------|------|-----------|---------|---------|
| 抹茶アイスクリーム120ml | PRD-101 | 1,500.00 | 個 | LOT-2026-03-M01 | 冷凍庫A-01 | 2026/09/30 |
| 抹茶アイスクリーム500ml | PRD-102 | 320.00 | 個 | LOT-2026-03-M02 | 冷凍庫A-01 | 2026/09/30 |
| ゆずソルベ | PRD-201 | 180.00 | 個 | LOT-2026-03-Y01 | 冷凍庫A-02 | 2026/08/15 |
| 土佐ジロー卵プリン | PRD-301 | 250.00 | 個 | LOT-2026-03-T01 | 冷蔵庫B-01 | 2026/03/25 |
| よさこいマンゴーパフェ | PRD-401 | 400.00 | 個 | LOT-2026-03-K01 | 冷凍庫A-03 | 2026/10/31 |
| 四万十栗モンブランアイス | PRD-501 | 60.00 | 個 | LOT-2026-02-S01 | 冷凍庫A-03 | 2026/07/20 |
| 塩けんぴアイスバー | PRD-601 | 800.00 | 本 | LOT-2026-03-E01 | 冷凍庫B-01 | 2026/11/15 |

#### 何をするか
1. 確認済みの発注に対して、在庫引当を行う
2. まず「ORD-20260318-001 イオン高知店」の注文を確認:
   - 抹茶アイスクリーム120ml × 200個 → 在庫1,500個 → **引当可能**（残: 1,300個）
   - ゆずソルベ × 150個 → 在庫180個 → **引当可能**（残: 30個）⚠️ 在庫残少
   - 土佐ジロー卵プリン × 100個 → 在庫250個 → **引当可能**（残: 150個）
3. 「ORD-20260318-003 道の駅やす」の注文を確認:
   - 四万十栗モンブランアイス × 80個 → 在庫60個 → **在庫不足！** 20個不足
   - ゆずソルベ × 60個 → 在庫30個（イオン分引当後）→ **在庫不足！** 30個不足
4. 在庫不足が発生した場合:
   - 製造部門に増産依頼を出す（システム外の連絡）
   - または、得意先に納期調整の連絡を行う
   - 注文の一部引当（部分出荷）を行い、残りはバックオーダーとして管理
5. 引当完了した注文は「在庫引当済」ステータスに更新

### 4.2 受注（OCR読取）ページ（`/ocr`）

**表示すべき情報**:
- ページタイトル「AI-OCR 伝票読取」と説明文
- 画像アップロードエリア（ドラッグ&ドロップ対応）

- 最近のスキャン履歴テーブル（スキャンID / 文書種別 / ステータス / 信頼度 / スキャン日時）

**必要なアクション**:
- ファイルアップロード（ドラッグ&ドロップ / クリック選択）
- OCR処理の開始（自動）
- スキャン結果の確認（テーブル行クリック → 詳細表示）
- 失敗時の再スキャン
- 手動入力への切り替え

**ステータス表示**:
- PENDING: グレーバッジ
- PROCESSING: 青色バッジ（パルスアニメーション）
- COMPLETED: 緑色バッジ
- FAILED: 赤色バッジ

**アラート条件**:
- OCR信頼度が70%未満の場合、赤色で表示し手動確認を促す
- FAILED状態のスキャンが24時間以上放置されている場合、アラート通知

**操作のしやすさ**:
- アップロードエリアは画面の目立つ位置に大きく表示（`p-12`）
- 対応ファイル形式を明記: JPEG, PNG, GIF, WebP, TIFF
- ドラッグ中のビジュアルフィードバック（ボーダー色変化、背景色変化）

### 4.4 在庫管理ページ（`/inventory`）

**表示すべき情報**:
- ページタイトル「在庫管理」と説明文
- 検索バー（虫眼鏡アイコン付き）
- フィルター: 保管場所プルダウン / 期限状態プルダウン
- フィルター結果件数
- 在庫一覧テーブル（商品名 / 商品コード / 所有企業 / 数量 / ロット番号 / 保管場所 / 賞味期限）

**必要なアクション**:
- 「入庫登録」ボタンでモーダル表示（商品名 / 商品コード / 数量 / 所有企業 / ロット番号 / 保管場所 / 賞味期限）

- 数量調整（実在庫との差異修正）
- 検索・フィルタリング
- 安全在庫設定

**ステータス表示（賞味期限ベース）**:
- 期限切れ: 行全体赤色背景、「期限切れ」バッジ（赤）

- 在庫が0になった商品: 行全体を強調表示

**操作のしやすさ**:
- 検索はリアルタイムフィルタリング（入力と同時に結果更新）
- 数量表示は小数点2桁、単位付き
- 商品コードはモノスペースフォント、グレー背景バッジで視認性向上
- タブレット対応（フィルターのプルダウンは十分な高さ）

## 5. 非機能要件

### 5.1 パフォーマンス

| 項目 | 要件 |
|------|------|
| ページ初期表示 | 2秒以内（First Contentful Paint） |
| API応答時間 | 500ms以内（P95） |
| OCR処理時間 | A4画像1枚あたり30秒以内 |
| 一覧テーブル表示 | 100件のデータを1秒以内にレンダリング |
| 検索・フィルター応答 | 入力から300ms以内に結果反映 |
| PDF生成 | 5秒以内（請求書1枚） |

### 5.2 レスポンシブ対応

| デバイス | 画面幅 | 対応レベル |
|---------|--------|-----------|
| デスクトップ | 1280px以上 | フル機能 |
| タブレット | 768px〜1279px | フル機能（レイアウト最適化） |
| スマートフォン | 〜767px | 主要機能のみ（閲覧・ステータス更新） |

- ワークフローステップバー: デスクトップ=横並び、モバイル=縦スタック
- テーブル: デスクトップ=通常表示、モバイル=横スクロール対応
- ナビゲーション: デスクトップ=横並び、モバイル=ハンバーガーメニュー

### 5.3 アクセシビリティ

| 項目 | 要件 |
|------|------|
| WAI-ARIA | ナビゲーション要素にaria-label属性（例: `aria-label="Workflow steps"`） |
| キーボード操作 | 全操作がキーボードのみで完結可能 |
| カラーコントラスト | WCAG 2.1 AA準拠（コントラスト比4.5:1以上） |
| フォーカス表示 | フォーカスリング（`focus-visible:outline`）の明示 |
| スクリーンリーダー | 装飾アイコンには `aria-hidden="true"`、ステータスバッジは読み上げ対応 |
| テキストサイズ | 最小14px（`text-sm`）、主要テキスト16px以上 |

### 5.4 セキュリティ

| 項目 | 要件 |
|------|------|
| 認証 | ユーザー名・パスワード認証（将来的にSSO対応） |
| 権限管理 | ロールベース: 事務担当者 / 管理職 / 倉庫担当 / 閲覧のみ |
| データ暗号化 | 通信: TLS 1.3、保存: AES-256 |
| セッション管理 | 30分無操作でセッションタイムアウト |
| 監査ログ | 全操作の操作者・日時・内容を記録 |

### 5.5 可用性

| 項目 | 要件 |
|------|------|
| 稼働時間 | 月〜金 7:00〜22:00（99.5%以上） |
| バックアップ | 日次自動バックアップ（30日間保持） |
| 障害復旧 | RTO: 4時間以内、RPO: 1時間以内 |

### 5.6 拡張性

- マルチテナント対応（将来的に他の食品企業へも展開可能な設計）
- API設計はRESTful、将来的にGraphQL対応も検討
- 外部サービス連携: 会計ソフト（freee、マネーフォワード）、配送業者（ヤマト運輸、佐川急便）のAPI連携

## 6. 用語集

| 用語 | 読み | 英語 | 説明 |
|------|------|------|------|
| 受注 | じゅちゅう | Order Receipt | 得意先からの注文を受け付けること |
| 発注 | はっちゅう | Purchase Order | 仕入先に対して商品を注文すること。本システムでは得意先からの注文（受注）をベースに管理 |
| OCR | - | Optical Character Recognition | 画像から文字を自動認識する技術。FAX画像から注文内容を読み取るために使用 |
| 在庫引当 | ざいこひきあて | Stock Allocation | 受注に対して在庫を確保（予約）すること。引当済みの在庫は他の注文に使えない |
| 安全在庫 | あんぜんざいこ | Safety Stock | 欠品を防ぐために最低限確保しておく在庫数量 |
| バックオーダー | - | Back Order | 在庫不足により即時出荷できない注文。入荷次第出荷する |
| ロット番号 | - | Lot Number | 製造単位を識別する番号。同一ロットは同じ製造条件で作られた製品群 |
| 賞味期限 | しょうみきげん | Best Before Date | 品質が保持される期限。食品業界では先入先出（FIFO）の原則で管理 |
| 赤伝票 | あかでんぴょう | Credit Note | 返品や値引きにより、元の売上伝票を取り消すための伝票 |
| 仕訳 | しわけ | Journal Entry | 取引を借方・貸方に分けて記録する会計処理 |
| 売掛金 | うりかけきん | Accounts Receivable | 商品を販売したが代金未回収の債権 |
| 軽減税率 | けいげんぜいりつ | Reduced Tax Rate | 食品等に適用される消費税率8%（標準税率10%に対して） |
| ピッキング | - | Picking | 出荷指示に基づき、倉庫から商品を取り出す作業 |
| 得意先 | とくいさき | Customer / Client | 継続的に取引のある販売先企業・店舗 |
| 冷凍庫 | れいとうこ | Freezer Storage | -18℃以下で保管する倉庫。アイスクリーム等の保管に使用 |
| 冷蔵庫 | れいぞうこ | Refrigerated Storage | 0〜10℃で保管する倉庫。プリン等の保管に使用 |
| DX | - | Digital Transformation | デジタル技術を活用した業務変革 |
| ワークフロー | - | Workflow | 業務の一連の流れ。本システムでは受注→確認→在庫→出荷→請求→返品の6ステップ |
| インライン編集 | - | Inline Editing | 一覧画面上で直接データを編集できるUI方式。別画面への遷移が不要 |
| インクリメンタルサーチ | - | Incremental Search | 文字を入力するたびにリアルタイムで検索結果が絞り込まれる方式 |
| 先入先出 | さきいれさきだし | FIFO (First In, First Out) | 先に入庫した商品から先に出荷する原則。食品業界での基本ルール |
